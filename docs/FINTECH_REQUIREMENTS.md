# FINTECH 商業級軟體開發需求規格（v2）

> 版本：v2.0  
> 更新日期：2026-03-03  
> 適用分支：`fintech`

---

## 1. 文件目的與適用範圍

本文件定義系統從課程作業版本升級為商業級 Fintech 系統的「可執行需求」。

### 1.1 目的

- 提供產品、工程、資安、法遵、測試、維運共同語言。
- 明確定義需求編號、驗收條件、量化指標與交付標準。

### 1.2 目前系統邊界

- 目前核心能力：投資/清算資料服務（以練習版資料模型承載）、RMI 遠端呼叫、手動交易、併發測試。
- 非本階段範圍：支付清算、實際 KYC 供應商串接、跨境法規在地化流程。

---

## 2. 名詞定義

- **ACID**：原子性、一致性、隔離性、持久性。
- **RBAC**：角色式存取控制（Role-Based Access Control）。
- **SLO**：服務等級目標（Service Level Objective）。
- **RPO/RTO**：資料可容忍遺失時間 / 系統復原時間目標。
- **PII**：可識別個人資訊。

---

## 3. 利害關係人與責任

- **PO**：定義需求優先順序與驗收口徑。
- **Engineering Lead**：架構決策、技術風險控管。
- **Security/GRC**：資安控制、法遵映射、稽核需求。
- **QA**：測試策略、測試覆蓋率與驗收報告。
- **SRE/DevOps**：部署、監控、告警、故障處置。

## 4. 功能需求（FR）與驗收條件（AC）

## FR-001 使用者身分驗證與授權

### 需求

- 系統需支援登入驗證（帳密 + MFA 預留）。
- 系統需支援 RBAC 權限檢查。

### AC

- AC-001：未登入不得執行任何寫入操作。
- AC-002：無權限角色執行寫入操作時，回傳 `403/AccessDenied`（若仍為 RMI，需回傳對應業務錯誤碼）。
- AC-003：授權失敗事件需寫入稽核日誌。

## FR-002 核心金融資料 CRUD（業務核心）

### 需求

- 提供核心領域資料 `Create/Read/Update/Delete`（例如 Portfolio/Holding/Payment）。
- 請求需包含操作者識別（userId）與請求追蹤 ID（requestId）。

### AC

- AC-004：CRUD 所有 API/服務方法有一致輸入驗證。
- AC-005：寫入成功返回可追蹤資訊（transactionId 或 requestId）。
- AC-006：無效輸入（空值、長度超限、格式錯誤）應被拒絕並記錄。

## FR-003 交易管理與錯誤補償

### 需求

- 所有寫入流程以手動交易處理。
- 成功 `commit`；任一錯誤 `rollback`。

### AC

- AC-007：重複主鍵/約束錯誤時資料不可部分寫入。
- AC-008：rollback 失敗需有錯誤告警與事件記錄。
- AC-009：交易結果可由測試驗證為一致狀態。

## FR-004 併發處理

### 需求

- 支援多客戶端同時讀寫。
- 可辨識併發衝突（busy/locked）並可觀測。

### AC

- AC-010：併發測試報告需含成功率、衝突率、平均延遲。
- AC-011：衝突事件需可追蹤到操作類型與請求 ID。

## FR-005 稽核日誌

### 需求

- 關鍵事件必須可稽核：登入、授權失敗、資料異動、交易失敗、管理員操作。

### AC

- AC-012：每筆稽核事件至少包含時間、操作者、動作、目標、結果、requestId。
- AC-013：日誌不可被一般業務帳號修改。

## 5. RBAC 權限矩陣（最小可行版）

| 角色       | 查詢核心資料 | 新增核心資料 | 修改核心資料 | 刪除核心資料 | 查看稽核 |
| ---------- | -------: | -------: | -------: | -------: | -------: |
| Viewer     |       ✅ |       ❌ |       ❌ |       ❌ |       ❌ |
| Operator   |       ✅ |       ✅ |       ✅ |       ❌ |       ❌ |
| Supervisor |       ✅ |       ✅ |       ✅ |       ✅ | ✅(只讀) |
| Admin      |       ✅ |       ✅ |       ✅ |       ✅ |       ✅ |

規則：

- 預設拒絕（Deny by default）。
- 任何新增權限需經 PR + 資安審核。

## 6. 資料與稽核欄位規格

## 6.1 業務資料（核心領域）

- `portfolio_id`：主鍵，不可重複，不可空。
- `asset_symbol`：不可空，格式需驗證（例如股票代碼）。
- `payment_id`：支付清算主鍵，不可重複。

> 備註：`EMP` 可作為訓練資料集，但不應被視為最終業務核心模型。

## 6.2 稽核事件欄位（AuditEvent）

- `eventId`（UUID）
- `eventTime`（UTC）
- `actorId`
- `actorRole`
- `action`（CREATE/READ/UPDATE/DELETE/LOGIN/...）
- `targetType`（EMP/USER/...）
- `targetId`
- `result`（SUCCESS/FAIL）
- `errorCode`（可空）
- `requestId`（跨服務追蹤）
- `sourceIp`（若可取得）

## 7. 非功能需求（NFR）與量化 SLO

## NFR-001 可用性

- 月可用性目標：**99.9%**。
- 單月不可用時間上限：約 43 分鐘。

## NFR-002 效能（初期目標）

- 讀取請求：`p95 < 500ms`、`p99 < 1000ms`。
- 寫入請求：`p95 < 800ms`、`p99 < 1500ms`。

## NFR-003 一致性與可靠性

- 寫入路徑必須具備交易原子性。
- RPO ≤ 15 分鐘，RTO ≤ 60 分鐘（v2 目標）。

## NFR-004 擴展性

- 併發客戶端 20 內需維持功能正確性與資料一致性。
- 若達到 SQLite 寫入瓶頸，需提出 DB 升級計畫。

## NFR-005 可觀測性

- 必備指標：QPS、錯誤率、延遲分位數、交易失敗數、鎖衝突數。
- 必備告警：錯誤率 > 2%、p95 超標連續 5 分鐘、服務不可用。

## 8. 安全需求（SEC）

## SEC-001 身分與憑證

- 密碼需雜湊儲存（禁用明文）。
- 服務金鑰與 DB 憑證不得寫死於程式碼。

## SEC-002 傳輸與儲存加密

- 內外部通訊需 TLS（正式環境）。
- 敏感資料需加密儲存（正式環境）。

## SEC-003 輸入驗證與防護

- 所有輸入做白名單驗證。
- 嚴禁動態拼接 SQL（必須 parameterized query）。

## SEC-004 稽核與追蹤

- 安全事件需 5 分鐘內可查。
- 稽核資料保存期限至少 1 年（依法規調整）。

## SEC-005 安全測試門檻

- SAST：不得存在 Critical/High 未豁免漏洞。
- SCA：不得存在 Critical 依賴漏洞。

---

## 9. 法遵需求（COMP）

> 實際適用依上線地區決定，本文件先定義控制框架。

## COMP-001 隱私與資料主體權利

- 需可回應資料查詢、修正、刪除（依法規與業務留存政策）。

## COMP-002 稽核可追溯

- 所有高風險操作需具不可否認證據（操作者 + 時間 + 內容）。

## COMP-003 變更治理

- 任何生產變更需經雙人審查與 CI 驗證。
- 緊急變更需補齊事後審核紀錄。

## 10. 架構與技術演進要求

## Phase 1（現況）

- Java RMI + SQLite
- 手動交易 + 併發驗證

## Phase 2（短期）

- 將服務契約抽象，便於替換為 REST/gRPC
- 導入統一錯誤碼、requestId、結構化日誌
- 建立 CI 安全掃描與品質閘門

## Phase 3（中期）

- DB 升級 PostgreSQL（支援更佳併發與 HA）
- 導入快照備份、災難復原演練
- 完成法遵證據鏈（稽核、報表、追溯）

---

## 11. 測試策略與交付門檻

## TEST-001 單元測試

- 業務邏輯 line coverage ≥ 70%。

## TEST-002 整合測試

- 覆蓋 CRUD、交易 rollback、授權失敗、錯誤碼驗證。

## TEST-003 併發測試

- 測試組態需固定（客戶端數、每客戶端操作數、操作比例）。
- 報告需含：成功率、衝突率、最終資料一致性檢查。

## TEST-004 回歸測試門檻（Release Gate）

- `RmiTest`、`TransactionTest`、`ConcurrentTest` 全綠。
- 無阻斷級（Blocker）缺陷。

---

## 12. DevSecOps 與分支治理

- 分支策略：`main`（穩定）、`project`（作業基線）、`fintech`（商業化）。
- PR 必要檢查：
  1.  compile success
  2.  test success
  3.  SAST/SCA pass
  4.  Code Review 2 人核准（高風險變更）
- 版本策略：SemVer（`MAJOR.MINOR.PATCH`）。

## 14. 里程碑與時程（建議）

- **M1（1 週）**：需求定稿、錯誤碼規範、RBAC 定義。
- **M2（2 週）**：稽核日誌與 requestId 導入、測試補齊。
- **M3（2 週）**：安全掃描閘門、監控告警落地。
- **M4（2~4 週）**：DB 升級 PoC（SQLite → PostgreSQL）。

---

## 15. 開放決策事項

1. 長期通訊協定：RMI 保留或全面轉 REST/gRPC。
2. 部署模型：on-prem / cloud / hybrid。
3. 法遵地區：最終上線法域與稽核標準。
4. 是否導入 MFA 為上線前硬性條件。

---
